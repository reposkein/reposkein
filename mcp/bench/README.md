# RepoSkein Retrieval Benchmark (Track 1)

**Contents:** [Method](#method) · [Results](#results) · [Honesty caveats](#honesty-caveats) · [How to run](#how-to-run) · [Add fixtures](#how-to-add-fixtures)

A deterministic, no-LLM harness that compares **RepoSkein's structural retrieval** against a **grep-based agent** on labeled code-navigation tasks. It measures retrieval quality (precision/recall/F0.5 vs independent hand-labeled ground-truth) and context-token efficiency.

> **What this does NOT measure:** end-task success (whether an agent actually fixes a bug). That requires a full agent loop and is Track 2 (SWE-bench style, future). This measures only the retrieval layer — which functions each strategy surfaces and how many context tokens it costs to do so.

---

## Method

### RepoSkein arm

Reuses the MCP's own retrieval pipeline:

1. `resolveTarget(store, repoIds, { name: seed })` — resolves the seed function by name
2. `assembleProfile(store, repoIds, target, 1)` — fetches direct callers (`upstream`) or callees (`downstream`)
3. Retrieved set = the neighbor function ids; token cost = `JSON.stringify(profile).length / 4`

This is equivalent to a single `get_context_profile` MCP call.

### Grep arm

1. `rg -n --no-heading --color=never` on the repo (excludes `.reposkein/`, `target/`, `node_modules/`, `dist/`, and `bench/` — so grep isn't penalised for matching the benchmark's own fixtures/README, which contain the query terms)
2. Each matched line is mapped to its enclosing Function node (the tightest function whose `start_line ≤ line ≤ end_line` in the parsed graph)
3. Retrieved set = all enclosing function ids found

### Token cost model (transparent)

Two models are reported — the **generous-to-grep** model is the headline so the comparison cannot be inflated in RepoSkein's favour:

| Model | What is counted |
|---|---|
| **fns (headline, generous to grep)** | rg output bytes + bytes of each matched function body |
| files (upper bound) | rg output bytes + full file bytes for every matched file |

Ratio = grep(fns, generous) ÷ RepoSkein. A ratio > 1 means RepoSkein surfaces the same callers with less context.

### Scoring

**F0.5** (precision-weighted, per [Cognition's retrieval eval](https://cognition.ai)): `β = 0.5` means precision counts 4× more than recall. Context pollution (false positives injected into an LLM's window) is worse than missing a caller, because the agent can iterate.

Scored against **independent hand-labeled ground-truth** (not generated by RepoSkein — verified by reading the source and grepping call sites manually).

## Results

Run on RepoSkein itself, 5 tasks (`bench/` excluded from the grep arm; caller/callee ground-truth includes test callers):

| task | kind | RepoSkein F0.5 | grep F0.5 | RepoSkein tok | grep tok (fns/files) | token ratio |
|---|---|---|---|---|---|---|
| callers-round2 | callers | 1.00 | 0.38 | 197 | 1144 / 9333 | 5.8× |
| callers-drop_dangling_edges | callers | 1.00 | 0.71 | 308 | 2457 / 5663 | 8.0× |
| callers-parent_of | callers | 1.00 | 0.56 | 198 | 2234 / 5606 | 11.3× |
| callees-index_tree | callees | 1.00 | 0.11 | 692 | 6006 / 18478 | 8.7× |
| lookup-meta_json | lookup | 1.00 | 0.38 | 95 | 412 / 8219 | 4.3× |

**Mean token ratio on structural tasks: 8.4× (grep ÷ RepoSkein, generous-to-grep)** — comfortably past the PRD §11 "≥3× vs grep" gate. RepoSkein resolves to exactly the right functions (F0.5 = 1.00) while grep drags in the definition, comments, and test assertions.

_Measures the retrieval layer (no LLM): which functions each strategy surfaces + the context tokens to do so. Not end-task success. RepoSkein's edge is structural/impact queries on non-trivial files; the `lookup` task (resolve a unique symbol) is the regime closest to grep parity — and even there grep reads more context here because the enclosing files are large._

## Honesty caveats

**Where grep is competitive (or better):**

- **Unique-symbol lookup in small repos:** if the symbol name is globally unique and the file is small, a grep-and-read-file approach surfaces the definition just as accurately. The `lookup-meta_json` task was designed to test this regime. In practice, `meta_json` is a unique name and grep finds it — but it also returns extra hits (comments, test references), which lowers precision (F0.5 = 0.38 vs RepoSkein's 1.00 even on this "grep-favorable" task in this repo). In a smaller, simpler repo the precision gap narrows.
- **Agent persistence:** a grep-based agent can iterate — narrow the pattern, open specific files. A single-shot grep is the lower bound. [Augment's SWE-bench analysis](https://www.augmentcode.com/) shows that with enough persistence, grep+agent is competitive on many task types. This benchmark measures one-shot retrieval, not multi-turn.
- **No semantic index needed:** grep requires no preprocessing. RepoSkein requires the repo to be indexed (`.reposkein/` present).

**Where RepoSkein's edge is clear:**

- **Structural/impact queries** ("which functions call X?"): grep finds lines containing the name but also finds the definition itself, comments, test assertions, and string literals — all counted as false positives. RepoSkein returns only functions with a recorded CALLS edge, filtered by direction.
- **Token efficiency:** even with the generous-to-grep model (reading only matched function bodies, not whole files), grep uses 6–12× more tokens on structural tasks. At whole-file granularity the ratio is 30–50×.
- **Non-trivial files:** the efficiency gap grows with file size. In a 1000-line file, grep forces the agent to read the entire file to see one call site; RepoSkein surfaces just the caller's summary.

## Track 2 — end-task benchmark (next, planned)

Track 1 (this harness) measures the **retrieval layer** in isolation. The
**proper, headline benchmark** is Track 2: an **end-task** comparison with a real
agent loop, measuring whether RepoSkein's leaner context actually improves
outcomes — not just retrieval precision.

Planned shape (see the scoping doc once written):

- **Tasks:** real GitHub issues / SWE-bench-Verified-style items, plus
  RepoSkein-favourable **impact/refactor** tasks on larger repos (where the
  research says structure beats grep+persistence; small SWE-bench repos
  understate the edge).
- **Arms (same agent + model, swap the retrieval toolset):** (A) RepoSkein MCP
  tools, (B) grep/ripgrep-only (Claude-Code style), (C) optional embedding-RAG.
- **Metrics:** task **resolve-rate**, **total tokens to resolution**, and
  **tool-calls/turns** — reported together (never a single metric).
- **Hygiene:** qualitative "vibe-check" passes on 5–10 examples before
  quantitative runs; precision-weighted retrieval scoring carried over from
  Track 1.

Track 2 requires LLM/API budget and is non-deterministic, so it lives outside CI
as an opt-in run. It will be scoped in a dedicated design doc and built once
this Track-1 harness is stable.

## How to run

Prerequisites:
- `ripgrep` (`rg`) on PATH — `brew install ripgrep` on macOS
- The target repo must be indexed: `reposkein-indexer index <repoPath>` (writes `.reposkein/` at the repo root)

```bash
cd mcp
npm install          # picks up tsx devDep
npm run bench -- /path/to/repo
```

The harness exits with code 2 and a clear message if `rg` or `.reposkein/nodes.jsonl` is missing.

---

## How to add fixtures

1. Open `bench/fixtures/reposkein.json`
2. Add a task object. The `relevant` list must be **hand-labeled** by reading the source — do not use RepoSkein's own output as ground-truth.
3. Function names must be unique in the repo, or extend the schema to `name@arity` and update the resolver.
4. Run `npm run bench` and paste the updated table into this README.

Task kinds supported: `callers`, `callees`, `lookup`.

---

## What is excluded from the npm tarball

`bench/` is not listed in `package.json`'s `files` field and is not compiled by `tsc` (only `src/` is in `tsconfig.json`'s `include`). It is a development/measurement tool only.
